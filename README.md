# Учёт внутренних заявок

Небольшое, но «взрослое» приложение для внутренних заявок: сотрудник создаёт заявку, её можно
найти, отфильтровать, поменять статус, а администратор — удалить. Сделано как fullstack-связка
**FastAPI + React (TypeScript) + SQLite**.

Это тестовое задание, но писал я его так, как писал бы рабочий сервис: со слоями, тестами,
миграциями, Docker и CI. Ниже — как запустить за одну минуту и как всё устроено.

---

## Запустить за минуту

Нужен только Docker:

```bash
docker compose up --build
```

- Приложение — http://localhost:5173
- API и Swagger — http://localhost:8000/docs
- Войти администратором: логин `admin`, пароль `admin` (нужен только чтобы удалять заявки)

Хочется без Docker — тоже просто. Backend:

```bash
cd backend
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements-dev.txt
uvicorn app.main:app --reload
```

Frontend (в соседнем терминале):

```bash
cd frontend
npm install
npm run dev
```

Vite сам проксирует `/api` на бэкенд, так что ничего дополнительно настраивать не нужно.

---

## Что умеет

- **Создавать заявки** с валидацией: заголовок 3–120 символов, описание до 1000, приоритет.
- **Искать** по заголовку и описанию — без учёта регистра, и для русского текста тоже
  (это, кстати, отдельная история — см. ниже).
- **Фильтровать** по статусу и приоритету, **сортировать** по дате и приоритету, листать
  **постранично**. Всё это считает backend, а не браузер.
- **Менять статус** прямо в строке списка.
- **Удалять** заявки — только администратору.
- Соблюдать правила: заявку в статусе `done` нельзя ни редактировать, ни удалять, ни вернуть
  обратно в работу. На попытку приходит понятная ошибка, а не «500».
- Показывать честные состояния интерфейса: загрузка, пустой список, ошибка сервера.

## Как это устроено

**Backend** разложен по слоям, каждый занимается своим делом:

```
роутер  →  сервис (бизнес-правила)  →  репозиторий (SQL)  →  модель (таблица)
```

Роутер только принимает запрос, правило «`done` неизменяем» живёт в сервисе, а весь SQL
(фильтры, поиск, сортировка, пагинация) — в репозитории. Поверх — Pydantic-валидация, JWT для
админа, единый формат ошибок и Alembic-миграции.

**Frontend** организован по Feature-Sliced Design: `shared → entities → features → widgets`.
Серверное состояние держит TanStack Query — отсюда «бесплатные» состояния загрузки/ошибки/пустоты
и автоматическое обновление списка после изменений. Весь дизайн — на токенах (CSS-переменных),
менять палитру можно из одного файла.

### Одна история про поиск

SQLite умеет искать без учёта регистра только для латиницы — «Принтер» по запросу «принтер»
не нашёлся бы. Я это поймал тестом и починил: зарегистрировал свою функцию приведения к нижнему
регистру (`py_lower`), которая корректно работает и с кириллицей. Мелочь, но из таких мелочей и
складывается «работает по-настоящему».

## Тесты и качество

```bash
# Backend — pytest
cd backend && source .venv/bin/activate && pytest      # 48 тестов

# Frontend — vitest
cd frontend && npm run test                            # 16 тестов
```

Тесты писались до кода — это видно по истории коммитов. Линтеры (`ruff`, `eslint`), проверка
типов (`tsc`) и сборка гоняются на каждый push через GitHub Actions —
[`.github/workflows/ci.yml`](.github/workflows/ci.yml).

## API вкратце

| Метод    | Путь                | Что делает                                  | Кто     |
| -------- | ------------------- | ------------------------------------------- | ------- |
| `POST`   | `/api/auth/login`   | Вход, отдаёт JWT                            | —       |
| `GET`    | `/api/tickets`      | Список + фильтры/поиск/сортировка/страницы  | —       |
| `POST`   | `/api/tickets`      | Создать заявку                              | —       |
| `GET`    | `/api/tickets/{id}` | Одна заявка                                 | —       |
| `PATCH`  | `/api/tickets/{id}` | Изменить поля или статус                    | —       |
| `DELETE` | `/api/tickets/{id}` | Удалить заявку                              | админ   |

Параметры списка: `status`, `priority`, `q`, `sort_by` (`created_at` \| `priority`),
`order` (`asc` \| `desc`), `page`, `page_size`.
