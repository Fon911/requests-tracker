# Учёт внутренних заявок

Небольшое приложение для внутренних заявок. Сотрудник заводит заявку, её можно найти,
отфильтровать, отсортировать, поменять статус, а админ ещё и удалить. Стек: **FastAPI +
React (TypeScript) + SQLite**.

Формально это тестовое, но писал я его как обычный рабочий сервис: слои, тесты, миграции,
Docker, CI. Ниже как запустить и как всё внутри устроено.

## Запуск

Нужен только Docker. Копируешь, выполняешь, готово:

```bash
git clone https://github.com/Fon911/requests-tracker.git
cd requests-tracker
docker compose up --build
```

Что открыть:

- приложение — http://localhost:5173
- API и Swagger — http://localhost:8000/docs
- вход админа: `admin` / `admin` (нужен только чтобы удалять)

При первом запуске база сама наполняется демо-заявками с разными статусами, приоритетами и
датами, чтобы сразу было что посмотреть: список, фильтры, сортировку, пагинацию. Повторно на
непустой базе не сеётся, в `production` не сеётся вообще.

Если без Docker, тоже недолго. Бэкенд:

```bash
cd backend
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements-dev.txt
uvicorn app.main:app --reload
```

Фронт в соседнем терминале:

```bash
cd frontend
npm install
npm run dev
```

Vite сам проксирует `/api` на бэкенд, так что настраивать ничего не надо.

## Что умеет

- Заводить заявки с валидацией: заголовок 3–120 символов, описание до 1000, приоритет.
- Искать по заголовку и описанию, без учёта регистра и по-русски тоже (про это ниже есть
  отдельная история).
- Фильтровать по статусу и приоритету, сортировать по дате и приоритету, листать постранично.
  Всё это считает бэкенд, а не браузер.
- Менять статус прямо в строке списка.
- Удалять заявки, но только админу.
- Держать правило: заявку в статусе `done` нельзя ни редактировать, ни удалять, ни вернуть в
  работу. На попытку прилетает понятная ошибка, а не голый «500».
- Показывать честные состояния: загрузка, пустой список, ошибка сервера.

## Как устроено

Бэкенд разложен по слоям, каждый занят своим:

```
роутер  →  сервис (бизнес-правила)  →  репозиторий (SQL)  →  модель (таблица)
```

Роутер только принимает запрос. Правило «`done` не трогаем» живёт в сервисе. Весь SQL (фильтры,
поиск, сортировка, пагинация) сидит в репозитории. Сверху Pydantic-валидация, JWT для админа,
единый формат ошибок, миграции через Alembic.

Фронт сделан по Feature-Sliced Design: `shared → entities → features → widgets`. Серверным
состоянием рулит TanStack Query, отсюда почти бесплатные состояния загрузки/ошибки/пустоты и
авто-обновление списка после изменений. Дизайн на CSS-токенах, палитра меняется из одного файла.

### Одна история про поиск

SQLite умеет искать без учёта регистра только для латиницы, так что «Принтер» по запросу
«принтер» не нашёлся бы. Поймал это тестом и починил: зарегистрировал свою функцию приведения к
нижнему регистру (`py_lower`), которая нормально работает и с кириллицей. Мелочь, но из таких
мелочей и складывается «реально работает».

## Тесты

```bash
# бэкенд, pytest
cd backend && source .venv/bin/activate && pytest      # 49 тестов

# фронт, vitest
cd frontend && npm run test                            # 23 теста
```

Тесты писались раньше кода, это видно по истории коммитов. Линтеры (`ruff`, `eslint`), проверка
типов (`tsc`) и сборка гоняются на каждый push через GitHub Actions, конфиг в
[`.github/workflows/ci.yml`](.github/workflows/ci.yml).

## API вкратце

| Метод    | Путь                | Что делает                                 | Кто   |
| -------- | ------------------- | ------------------------------------------ | ----- |
| `POST`   | `/api/auth/login`   | Вход, отдаёт JWT                           | все   |
| `GET`    | `/api/tickets`      | Список + фильтры/поиск/сортировка/страницы | все   |
| `POST`   | `/api/tickets`      | Создать заявку                             | все   |
| `GET`    | `/api/tickets/{id}` | Одна заявка                                | все   |
| `PATCH`  | `/api/tickets/{id}` | Поменять поля или статус                   | все   |
| `DELETE` | `/api/tickets/{id}` | Удалить заявку                             | админ |

Параметры списка: `status`, `priority`, `q`, `sort_by` (`created_at` или `priority`),
`order` (`asc` или `desc`), `page`, `page_size`.
